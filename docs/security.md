# Безопасность

## Модель безопасности

Competitive N-Back считает клиент недоверенной средой. Все действия, изменяющие игру, проверяются на сервере.

Клиент может запросить действие, но сервер решает:

- авторизован ли пользователь;
- состоит ли пользователь в комнате;
- находится ли комната в нужном состоянии;
- является ли ответ верным;
- является ли ответ повторным;
- должна ли игра завершиться.
- как должен измениться рейтинг после матча.

## Авторизация

Авторизация реализована через better-auth.

Protected-процедуры требуют активную сессию. `userId` из сессии используется для:

- владения комнатой;
- идентичности игрока;
- привязки счёта;
- фильтрации личной истории.

Клиент не должен передавать `userId` для защищённых игровых действий. Сервер берёт его из сессии.

## Server-Authoritative проверка ответов

Payload ответа содержит только:

```json
{
  "roomId": "room_123"
}
```

Сервер сам вычисляет:

- текущий индекс стимула;
- текущую позицию;
- ожидаемое совпадение;
- корректность ответа.

Это запрещает клиенту отправлять поддельные данные о правильности ответа.

## Защита от читов

| Угроза | Защита |
| --- | --- |
| Клиент локально меняет счёт | Счёт отображается только из серверного состояния. |
| Клиент отправляет фейковый `isCorrect` | API не принимает корректность от клиента. |
| Клиент отвечает за другого пользователя | `userId` берётся из сессии, а не из body. |
| Клиент присоединяется к запущенной игре | Сервер требует статус `lobby`. |
| Клиент запускает чужую комнату | Сервер проверяет `ownerId`. |
| Клиент спамит ответами | Сервер хранит факт ответа на каждый стимул. |

## Защита от спама

Спам ответами ограничивается через:

- один принятый ответ на игрока на стимул;
- серверное отклонение дублей;
- tRPC validation;
- rate limiting на уровне deployment-инфраструктуры.

Пример отклонения дубля:

```json
{
  "code": "BAD_REQUEST",
  "message": "Player already answered this stimulus."
}
```

## Валидация

Все входящие данные валидируются до изменения состояния.

Валидация создания комнаты:

- `n` должен быть `2`, `3` или `4`;
- `length` должен быть больше `n`;
- `baseIntervalMs` должен быть не меньше минимального интервала.

Валидация жизненного цикла:

- присоединение возможно только в `lobby`;
- старт требует владельца и 2-4 игроков;
- ответ возможен только в `running`;
- ручное завершение доступно владельцу;
- автоматическое завершение происходит при конце последовательности.

## Целостность данных

Правила записи в БД:

- ответы append-only;
- счёт обновляется только из результата серверной проверки;
- победитель записывается только при переходе комнаты в `finished`;
- рейтинг и ранг обновляются только сервером после финализации матча;
- удаление пользователя каскадно очищает связанные auth/session данные согласно схеме.
