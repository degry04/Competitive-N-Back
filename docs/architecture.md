# Архитектура

## Архитектурная модель

Competitive N-Back использует server-authoritative архитектуру. Клиент отвечает за отображение и ввод. Сервер отвечает за игровую истину.

Такой подход необходим, потому что несколько игроков делят одну игровую временную шкалу. Если каждый клиент самостоятельно рассчитывает текущий стимул или корректность ответа, задержки сети, расхождение часов и модификация клиента приведут к разным результатам у игроков.

## Роли

### Клиент

Клиент:

- Авторизует пользователя через better-auth.
- Создаёт комнаты, присоединяется к ним и запускает игру через tRPC mutations.
- Подписывается на обновления игрового состояния.
- Отображает сетку стимулов 3x3.
- Отправляет попытки ответа на сервер.
- Показывает счёт, штрафы и финальные результаты.

Клиент не должен:

- Генерировать последовательность стимулов.
- Решать, верен ли ответ.
- Продвигать авторитетные игровые часы.
- Рассчитывать победителя.

### Сервер

Сервер:

- Создаёт комнаты и валидирует настройки.
- Хранит игроков и членство в комнате.
- Генерирует последовательность стимулов.
- Отслеживает текущий индекс и интервал.
- Проверяет каждый ответ.
- Запрещает повторные ответы на один стимул.
- Применяет штрафы и ускорение.
- Рассылает состояние подписанным клиентам.
- Сохраняет завершённые игры и результаты игроков.
- После завершения матча рассчитывает ELO-рейтинг и обновляет ранги игроков.

## Компоненты

### Frontend

Frontend построен на Next.js App Router. Он включает:

- Экраны входа и регистрации.
- UI создания и выбора комнаты.
- Игровую сетку.
- Таблицу результатов.
- Страницы истории.

Локальное состояние интерфейса выводится из серверных ответов и realtime-событий.

### Backend API

Backend API реализован через tRPC. Он предоставляет типизированные процедуры для:

- Жизненного цикла комнаты.
- Игровых действий.
- Realtime-подписок.
- Получения истории.

tRPC выбран потому, что frontend и backend написаны на TypeScript. Это даёт общий типизированный контракт без отдельной генерации REST-клиента.

### Game Engine

Игровой движок — отдельный TypeScript-модуль доменной логики. Он отвечает за:

- Генерацию последовательности.
- Определение N-back совпадений.
- Проверку ответа.
- Изменение счёта.
- Штрафную механику.
- Расчёт победителя.

Изоляция движка от UI и транспорта делает правила проверяемыми через Vitest и упрощает API-слой.

## Data Flow

### Создание комнаты

1. Клиент вызывает `room.create`.
2. Сервер валидирует настройки.
3. Сервер создаёт запись комнаты.
4. Сервер создаёт членство владельца.
5. Сервер возвращает snapshot комнаты.
6. Клиент выбирает созданную комнату.

### Подключение к комнате

1. Клиент вызывает `room.join`.
2. Сервер проверяет авторизацию.
3. Сервер проверяет, что комната в статусе `lobby`.
4. Сервер проверяет лимит игроков.
5. Сервер добавляет игрока.
6. Сервер рассылает `game:update`.

### Запуск игры

1. Владелец вызывает `room.start`.
2. Сервер проверяет права владельца.
3. Сервер проверяет, что игроков 2-4.
4. Сервер генерирует последовательность.
5. Сервер переводит комнату в `running`.
6. Сервер отправляет `game:start`.
7. Сервер начинает продвигать игровую временную шкалу.

### Отправка ответа

1. Клиент вызывает `game.submitAnswer`.
2. Сервер вычисляет текущий индекс стимула по серверному времени.
3. Сервер отклоняет повторный ответ того же игрока на тот же стимул.
4. Сервер проверяет, является ли текущий стимул совпадением.
5. Сервер обновляет счёт или ошибки.
6. Сервер ускоряет игру, если игрок достиг каждой третьей ошибки.
7. Сервер отправляет `game:update`.

### Завершение игры

1. Сервер достигает последнего стимула или владелец завершает игру вручную.
2. Сервер рассчитывает победителя.
3. Сервер рассчитывает рейтинг по pairwise ELO-сравнениям.
4. Сервер сохраняет игру, результаты и `rating_history`.
5. Сервер отправляет `game:end`.
6. Клиенты отображают итоговую таблицу и обновлённые рейтинги.

## Игровое состояние

| Поле | Назначение |
| --- | --- |
| `roomId` | Идентификатор общей multiplayer-сессии. |
| `status` | `lobby`, `running` или `finished`. |
| `n` | Количество шагов назад для сравнения. |
| `sequence` | Сгенерированный сервером массив позиций 0-8. |
| `currentIndex` | Текущий индекс стимула. |
| `currentPosition` | Текущая клетка сетки. |
| `baseIntervalMs` | Начальный интервал. |
| `currentIntervalMs` | Текущий интервал после штрафов. |
| `players` | Счёт, ошибки, штрафы и состояние ответов игроков. |
| `startedAt` | Серверное время запуска. |
| `finishedAt` | Серверное время завершения. |

## Почему такая архитектура

Server-authoritative модель даёт:

- Единое игровое время для всех игроков.
- Достоверный подсчёт очков.
- Защиту от клиентских читов.
- Один источник истины для сохранения результатов.
- Тестируемую доменную логику.

Realtime-подписки нужны потому, что polling добавляет задержку и делает быстрые раунды менее синхронными.
