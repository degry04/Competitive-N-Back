# Тестирование

## Стек

Competitive N-Back использует Vitest для unit- и domain-тестов.

Главная зона тестирования — игровой движок. Именно там находятся правила, которые должны оставаться стабильными независимо от изменений UI, API или realtime-слоя.

## Что тестируется

| Область | Цель |
| --- | --- |
| Генерация последовательности | Проверить, что позиции лежат в диапазоне клеток сетки. |
| N-back matching | Проверить определение совпадений для поддерживаемых `n`. |
| Проверка ответа | Убедиться, что верные и неверные ответы корректно меняют состояние. |
| Защита от дублей | Убедиться, что игрок не может дважды ответить на один стимул. |
| Штрафная логика | Проверить, что каждая третья ошибка ускоряет игру. |
| Расчёт победителя | Обеспечить детерминированную таблицу результатов. |

## Запуск тестов

```bash
npm test
```

## Примеры тест-кейсов

### Определяет 2-back совпадение

Input:

```ts
const sequence = [4, 1, 4];
const n = 2;
const index = 2;
```

Expected:

```ts
isMatchAt({ sequence, n }, index) === true;
```

### Отклоняет повторный ответ

Сценарий:

1. Игрок отвечает на стимул `5`.
2. Сервер сохраняет, что игрок уже ответил на стимул `5`.
3. Игрок отправляет ответ повторно до следующего tick.

Expected:

```text
Error: Player already answered this stimulus.
```

### Применяет штраф за неверный ответ

Начальное состояние:

```json
{
  "correct": 3,
  "errors": 2,
  "penalty": 2,
  "currentIntervalMs": 1600
}
```

После одного неверного ответа:

```json
{
  "correct": 3,
  "errors": 3,
  "penalty": 3,
  "currentIntervalMs": 1440
}
```

### Рассчитывает победителя

Игроки:

```json
[
  { "displayName": "Alice", "correct": 10, "errors": 3 },
  { "displayName": "Bob", "correct": 10, "errors": 1 },
  { "displayName": "Carol", "correct": 8, "errors": 0 }
]
```

Ожидаемый победитель:

```text
Bob
```

Bob имеет столько же верных ответов, сколько Alice, но меньше ошибок.

## Стратегия тестирования

Сначала покрывается чистая игровая логика. API-тесты должны проверять авторизацию и границы валидации. UI-тесты должны проверять пользовательские сценарии, не дублируя проверки движка.

Приоритеты покрытия:

1. Корректность правил игрового движка.
2. Валидация жизненного цикла комнаты.
3. Штрафы и изменение скорости.
4. Доступ к API только для авторизованных пользователей.
5. Порядок realtime-событий.
